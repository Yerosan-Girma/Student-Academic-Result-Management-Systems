# Contributing Guidelines

Thank you for your interest in contributing to the Student Academic Result Management System!

## How to Contribute

### 1. Fork the Repository
Click the "Fork" button on GitHub to create your own copy.

### 2. Clone Your Fork
```bash
git clone https://github.com/YOUR_USERNAME/Student-Academic-Result-Management-Systems.git
cd Student-Academic-Result-Management-Systems
```

### 3. Create a Feature Branch
```bash
git checkout -b feature/your-feature-name
```

### 4. Make Your Changes
- Follow the existing code style
- Add comments where necessary
- Test your changes thoroughly

### 5. Commit Your Changes
```bash
git commit -m "feat: description of your changes"
```

Use conventional commit messages:
- `feat:` for new features
- `fix:` for bug fixes
- `docs:` for documentation
- `style:` for code style changes
- `refactor:` for code refactoring
- `test:` for test updates

### 6. Push to Your Fork
```bash
git push origin feature/your-feature-name
```

### 7. Create a Pull Request
Go to the original repository and create a PR with a clear description.

---

## Code Standards

### Backend (Node.js/Express)
- Use ES6+ syntax
- Follow ESLint rules
- Add error handling
- Document API endpoints
- Write meaningful variable names

### Frontend (React)
- Use functional components with hooks
- Follow React best practices
- Add PropTypes or TypeScript
- Keep components small and reusable
- Use CSS modules or styled-components

### Database
- Use migrations for schema changes
- Add indexes for performance
- Use meaningful table/column names
- Document complex queries

---

## Testing

### Backend Testing
```bash
npm test
```

### Frontend Testing
```bash
npm run test
```

---

## Issues

### Reporting Bugs
- Describe the bug clearly
- Provide steps to reproduce
- Include screenshots/logs if applicable
- Specify your environment (OS, browser, etc.)

### Requesting Features
- Explain the use case
- Provide examples
- Discuss potential implementation approach

---

## Pull Request Process

1. Update documentation if needed
2. Run tests locally
3. Ensure code follows standards
4. Provide clear PR description
5. Link related issues if any
6. Wait for review and feedback

---

## Questions?

Feel free to open an issue or start a discussion if you have questions about contributing.

Happy contributing! 🚀
